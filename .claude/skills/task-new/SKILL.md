# タスクのセットアップ

このコマンドが呼び出されたら以下の指示に従って順番に進めてください

## ユーザー入力

```
$ARGUMENTS
```

処理を進める前に、ユーザー入力を必ず考慮してください(空でない場合)。
ユーザー入力が与えられていない場合はユーザーにタスクの概要を尋ねてください

## Step1: タスクのセットアップ

この Step は general-purpose サブエージェント(model: sonnet)で実行してください

### タスクのブランチを作成する

すでに適切なブランチ名でブランチが作成されている場合はこの手順をスキップしてください

1. 簡潔な短縮名(2~4 単語)を生成する

- 機能説明を分析し、最も意味のあるキーワードを抽出
- 機能の本質を捉えた英語で 2〜4 単語の短縮名を作成
- 技術用語や略語を保持(OAuth2、API、JWT など)
- 簡潔でありながら、一目で機能を理解できる程度に説明的に

2. 考えたブランチ名にタスクの性質に応じて新機能なら`feature/`、バグ修正なら`bugfix/`、リファクタなら`refactor/`の prefix をつけてブランチ名とする

3. ブランチを作成する
   git switch サブコマンドを使用して
   ベースブランチを**AskUserQuestion**ツールを使いユーザーに確認する。
   指定がなければ`dev`を使用する。
   dev をベースにする場合は git fetch で dev を最新の状態にする

### タスクのディレクトリを作成する

まずは`*/tasks`にマッチするタスクを格納するディレクトリを特定してください。なければ root 配下に tasks ディレクトリを作成してください。

```bash
find -L . -type d -name "tasks" -not -path "*/node_modules/*" -not -path "*/.git/*"
```

このディレクトリを以下`TASK_ROOT_DIR`とします

TASK_ROOT_DIR 内には`[タスク番号(先頭0埋め3桁)]\_[短縮名]`の形式で命名されたディレクトリが存在している

1. TASK_ROOT_DIR を検索し、最大のタスク番号 N を見つける（タスクが存在しない場合は N=0 とする）
   ```bash
   ls -1 TASK_ROOT_DIR | grep -E '^[0-9]{3}_' | sort -r | head -1
   ```
2. 新たなタスク番号として N+1 を使用する
3. TASK_ROOT_DIR に`$(printf "%03d" $((N+1)))\_[短縮名]`の形式でディレクトリを作成する

### spec.md の作成

作成したディレクトリ内に以下の形式 spec.md を作成し、[]の中身のみを埋めてください
他のセクションは追加しないでください

```md
# 仕様: [機能名]

**ブランチ名**: `[ブランチ名]`
**作成日**: [YYYY-MM-DD]

## 背景

## 概要
```

## Step2: 対話的に仕様策定

仕様は Step1 で作成した spec.md に書き込むこと
spec.md の場所が分からない場合は停止してください

### Phase1: 実装内容の深堀り

まずは実装内容についてユーザーに**AskUserQuestion**ツールを使ってインタビューしてください。
できるだけ自由入力ではなく選択肢を提示すること。

**質問項目**

- 仕様
- 技術詳細
- UI/UX
- その他なんでも

不明点が完全になくなるまで深く掘り下げること
必要に応じて関連するファイル、関数、データモデルを探索してください。

**ガイドライン:**

- プロダクトのユーザーが何を必要とし、なぜ必要かに焦点を当てる
- **【重要】具体的なコード(API、コード構造)は記載しないこと**
  - 概念モデリング、migration は OK
  - リファクタなどコードの改善が目的のタスクであれば記載してよい
- 開発者ではなく、ビジネス関係者向けに書く
- 考慮が漏れている点がないか注意する

### Phase2: 実装内容の記述

決まった仕様を spec.md に書き込んでください

### Phase3: フィードバックを受け取る

spec.md に書き込んだ内容について自身でレビューして不明点などがないかチェックしてください。
また、ユーザーにも**AskUserQuestion**ツールを使ってフィードバックがない確認してください。
フィードバックがあれば Phase1 に戻りそれを元に議論してください。

## 最終 Step: ユーザーへ次のステップの指示

spec.md を確認して、問題なければ task-plan または task-implement コマンドを実行するようにユーザーに促してください
